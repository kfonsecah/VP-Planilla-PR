# Phase 67 — UI Design Contract

> Visual and interaction contract for three new employee profile tabs: Planillas, Eventos Laborales, Documentos.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Reference implementation:** The Aguinaldo tab in `src/frontend/src/app/pages/employee/[id]/page.tsx` (lines 250–338) is the gold standard. All three new tabs must match or exceed its visual quality.

## Spacing Scale

Declared abstract values (standard multiples of 4):

| Token | Value | Tailwind | Usage |
|-------|-------|----------|-------|
| xs | 4px | gap-1 / p-1 | Icon-to-text gaps, badge inner padding |
| sm | 8px | gap-2 / p-2 | Inline element spacing, badge pill margins |
| md | 16px | gap-4 / p-4 | Default content padding |
| lg | 24px | gap-6 / p-6 | Modal inner padding |
| xl | 32px | gap-8 | Section separation |
| 2xl | 48px | — | Not used in tab context |

### Locked Project-Level Spacing Exceptions

The following values fall outside the standard {4, 8, 16, 24, 32, 48, 64} set. They are **locked exceptions** inherited from existing implemented components. Executor must match these values exactly to maintain visual consistency with the implemented codebase.

| Value | Tailwind | Where Used | Justification |
|-------|----------|-----------|---------------|
| 20px | `p-5` | Card content body (`<div class="p-5">`) | Locked from `ProfileSummaryTab.tsx` card content area — all new tab cards must use `p-5` to align with the existing profile tab visual rhythm |
| 20px | `px-5 py-4` | Card section header strip | Locked from `ProfileSummaryTab.tsx` lines 72–74 — all tab card headers use `px-5 py-4` |
| 12px | `py-3` | Table `<th>` and `<td>` vertical padding | Locked from `EmployeeTable.tsx` header/cell pattern — `px-4 py-3` is the project-wide table cell baseline |

> Note: `p-5` (20px) and `py-3` (12px) are intentional codebase-locked exceptions. They are NOT bugs in the spec. The executor must use them exactly — do not substitute `p-4` (16px) or `p-6` (24px) for card content, and do not substitute `py-2` or `py-4` for table cells.

Additional layout exceptions:
- Tab container outer margin-bottom: `mb-6` — matches EmployeeProfileTabs
- Document list row: `px-5 py-3` — matches the info-row pattern from ProfileSummaryTab

Maximum 2 weights declared for this phase.

| Role | Size | Weight | Line Height | Tailwind |
|------|------|--------|-------------|---------|
| Body | 14px | 400 (normal) | 1.5 | `text-sm` |

| Badge text | 11px | 700 (bold) | 1.0 | `text-[11px] font-bold` |

## Tab-Specific Interaction Contracts

### Tab: Planillas (EmployeePayrollsTab)

**Primary focal point:** Net salary column (`Salario Neto`) — displayed in accent green bold to draw the eye; it is the most decision-relevant value per row.

**Layout:** Single card with header strip + scrollable table. No secondary cards.

**Section header:**
- Icon: `DocumentTextIcon` (24/outline)
- Title: "HISTORIAL DE PLANILLAS"

**Table columns (left to right):**

| Column | Content | Alignment | Width |
|--------|---------|-----------|-------|
| Período | `formatDateDisplay(period_start)` — `formatDateDisplay(period_end)` | left | auto |
| Tipo | `period_type` string (e.g. "quincenal") — capitalize first letter | left | auto |
| Estado | Status badge (BORRADOR / APROBADA / PAGADA) | center | 120px |
| Salario Bruto | `formatCRC(gross_salary)` | right | 140px |
| Deducciones | `formatCRC(total_deductions)` | right | 140px |
| Salario Neto | `formatCRC(net_salary)` — **bold, green text for emphasis** | right | 140px |
| Horas | `total_hours ?? '—'` + "h" suffix | center | 80px |
| Ajuste | Show `AdjustedBadge` only when `is_manually_adjusted === true` | center | 80px |
| Acción | Download button (see below) | center | 80px |

**Net salary column styling:**
```
<td class="px-4 py-3 text-sm font-bold text-[#4A5D3A] dark:text-green-400 text-right">
```

**Download button — enabled state (APROBADA or PAGADA):**
```
<button class="flex items-center gap-1 px-2.5 py-1.5 text-xs font-normal text-[#4A5D3A] dark:text-green-400 border border-[#4A5D3A] dark:border-green-600 rounded-lg hover:bg-[#E7DCC1] dark:hover:bg-green-900/20 transition-colors">
  <ArrowDownTrayIcon class="w-3.5 h-3.5" />
  Descargar
</button>
```

**Download button — disabled state (BORRADOR):**
```
<span class="flex items-center gap-1 px-2.5 py-1.5 text-xs text-zinc-300 dark:text-zinc-600 cursor-not-allowed" title="Solo disponible para planillas aprobadas o pagadas">
  <ArrowDownTrayIcon class="w-3.5 h-3.5" />
  Descargar
</span>
```

Interaction: calls `http.raw()` to fetch PDF blob with Bearer token, creates `URL.createObjectURL(blob)`, triggers `<a>.click()` — NOT `window.open()` (would fail auth). Confirmed from RESEARCH.md assumption A2 resolution.

**Manually adjusted badge:**
```
<span class="inline-flex items-center px-2 py-0.5 rounded-full text-[11px] font-bold bg-amber-100 text-amber-700 dark:bg-amber-900/30 dark:text-amber-300">
  Ajustado
</span>
```

Show in Ajuste column only when `is_manually_adjusted === true`, empty otherwise.

**Empty state — no payrolls:**
- Icon: `DocumentTextIcon`
- Heading: "Sin planillas registradas"
- Body: "Este empleado aún no ha participado en ninguna planilla."
- No CTA button (users cannot create payrolls from employee profile)

**Error state:**
- Icon: `ExclamationTriangleIcon` (red-400)
- Heading: "No se pudo cargar el historial de planillas"
- Body: "{error message} — Intenta de nuevo."
- Button: "Reintentar" (same pattern as page.tsx error state)

**Loading state:** 5 skeleton rows inside the table body

### Tab: Eventos Laborales (EmployeeEventsTab)

**Primary focal point:** Event name column (`Evento`) — leftmost and widest column; it is the primary identifier the user scans to find a specific assignment.

**Layout:** Single card with header strip + scrollable table. "Asignar Evento" button in the card header right side.

**Section header:**
- Icon: `CalendarDaysIcon` (24/outline)
- Title: "EVENTOS LABORALES"
- Right action: "Asignar Evento" primary button (opens LaborEventModal)

**Table columns:**

| Column | Content | Alignment |
|--------|---------|-----------|
| Evento | `labor_event_name` | left |
| Inicio | `formatDateDisplay(start_date)` | left |
| Fin | `formatDateDisplay(end_date)` or "—" if null | left |
| Estado | Status badge (active / completed / cancelled) | center |
| Acción | Delete icon button | center |

**Status badge values:**
- `active` → "Activo" — green badge
- `completed` → "Completado" — blue badge
- `cancelled` → "Cancelado" — red badge

**Delete action behavior:**
1. Click `TrashIcon` button
2. `window.confirm('¿Eliminar la asignación de este evento para este empleado?')` — matches LaborEventModal.tsx pattern (line 207)
3. On confirm: call `laborEventsService.deleteAssignment(assignmentId)`, show `toast.success('Evento eliminado')`, refresh list
4. On error: `toast.error('No se pudo eliminar el evento')`

**LaborEventModal reuse strategy:**
- Pass `employees={[currentEmployee]}` (single-element array) so the employee selector shows only the current employee pre-selected
- The form's `reset()` in `EmployeeEventsTab` must pre-set `employee_id` to `currentEmployee.id`
- The modal's employee selector is still visible but shows only one option (simpler than hiding it; hiding would require modal modification)

**Empty state — no events:**
- Icon: `CalendarDaysIcon`
- Heading: "Sin eventos laborales"
- Body: "Este empleado no tiene eventos laborales asignados."
- CTA: "Asignar Evento" primary button (same as header action)

**Error state:**
- Icon: `ExclamationTriangleIcon`
- Heading: "No se pudo cargar los eventos"
- Body: "{error} — Intenta de nuevo."
- Button: "Reintentar"

**Loading state:** 4 skeleton rows

### Tab: Documentos (EmployeeDocumentsTab)

**Primary focal point:** Document name (`file_path`) — the primary text in each list row; rendered at 14px/400 in zinc-700 so it reads as the primary label, with type and date rendered below at 12px/400 as secondary metadata.

**Layout:** Single card with header strip + list of document rows. "Agregar Documento" button in card header right side.

**Section header:**
- Icon: `FolderOpenIcon` (24/outline)
- Title: "DOCUMENTOS"
- Right action: "Agregar Documento" primary button (opens inline modal)

**Document list row layout** (not a full table — simpler list):

```
<div class="flex items-center justify-between px-5 py-3 border-b border-zinc-100 dark:border-zinc-800 last:border-0 hover:bg-zinc-50 dark:hover:bg-zinc-800/50 transition-colors">
  <div class="flex items-center gap-3">
    <DocumentIcon class="w-5 h-5 text-zinc-400 dark:text-zinc-500 flex-shrink-0" />
    <div>
      <p class="text-sm text-zinc-700 dark:text-zinc-200">{file_path}</p>
      <p class="text-xs text-zinc-400 dark:text-zinc-500">{document_type} · {formatDateDisplay(uploaded_at)}</p>
    </div>
  </div>
  [delete button]
</div>
```

**Why a list instead of a table:** Document metadata is limited to name + type + date. A table would create wasted whitespace. The list pattern scales better with long filenames and matches the "información personal" info-row pattern from ProfileSummaryTab.

**Document type display:** Render as-is from `document_type` field (no badge required — plain secondary text is sufficient for metadata-only documents)

**Delete action behavior:**
1. Click `TrashIcon` button
2. `window.confirm('¿Eliminar el documento "' + file_path + '"? Esta acción no se puede deshacer.')`
3. On confirm: `employeeDocumentService.delete(employeeId, docId)`, `toast.success('Documento eliminado')`, refresh list
4. On error: `toast.error('No se pudo eliminar el documento')`

**"Agregar Documento" modal:**

This is a new inline modal (NOT LaborEventModal) — simpler form with 2 fields:

- Modal header background: `bg-[#4A5D3A] dark:bg-green-700` — matches LaborEventModal green header
- Modal header title: "Agregar Documento"
- Modal width: `max-w-md`
- Modal shape: `rounded-2xl` — matches LaborEventModal
- Animation: `AnimatePresence` + `motion.div` with `scale 0.95→1, opacity 0→1, y 10→0` — matches LaborEventModal exactly (spring, damping 30, stiffness 400)
- Backdrop: `bg-black/60 backdrop-blur-sm z-40` — exact match

**Form fields (Agregar Documento modal):**

| Field | Label | Type | Validation |
|-------|-------|------|------------|
| `file_path` | Nombre del documento | text input | required, min 1 char, max 255 chars |
| `document_type` | Tipo de documento | text input (or select) | required, min 1 char, max 50 chars |

Input classes (exact match from LaborEventModal line 202):
```
w-full rounded-lg border border-zinc-200 dark:border-zinc-700 px-3 py-2.5 text-sm bg-zinc-50 dark:bg-zinc-800 text-zinc-700 dark:text-zinc-200 placeholder-zinc-400 focus:ring-2 focus:ring-green-500 focus:border-green-500 outline-none transition-all
```

Label classes (exact match):
```
block text-sm font-normal text-zinc-700 dark:text-zinc-300 mb-1.5
```

Error text under field:
```
<p class="mt-1 text-xs text-red-500 dark:text-red-400">{error message}</p>
```

**Modal footer:**
- Left: empty (no delete action for new-only modal)
- Right: "Cancelar" ghost button + "Guardar" primary button

"Guardar" button:
```
<button class="px-6 py-2.5 bg-green-600 hover:bg-green-700 text-white text-sm font-bold rounded-xl transition-colors shadow-sm disabled:opacity-50 disabled:cursor-not-allowed flex items-center gap-2">
  [Loading spinner when isSubmitting]
  Guardar documento

```

Loading spinner (inside button, matches LaborEventModal):
```
<div class="w-4 h-4 border-2 border-white border-t-transparent rounded-full animate-spin" />
Guardando...
```

**Keyboard behavior:** Escape key closes modal. First input focused on open. `useEffect` resets form and focuses first input when `isOpen` changes (matches CLAUDE.md constraint).

**Empty state — no documents:**
- Icon: `FolderOpenIcon`
- Heading: "Sin documentos"

- CTA: "Agregar Documento" primary button
