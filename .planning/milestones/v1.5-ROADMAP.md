## Phase Details

### Phase 32 — Schema: Capa de Ajustes + Campos Aprobación Planilla
**Goal:** Crear la tabla `vpg_clock_log_adjustments` para correcciones no destructivas y extender `vpg_payrolls` con campos de aprobación.

**Entregables:**
- Migración Prisma: tabla `vpg_clock_log_adjustments` con campos:
  - `adjustment_id` (PK)
  - `adjustment_clock_log_id` (FK nullable — null si es marca nueva)
  - `adjustment_employee_id` (FK)
  - `adjustment_type`: `add` | `edit` | `void` (agregar, editar, anular)
  - `adjustment_original_timestamp` (el valor original, null si es marca nueva)
  - `adjustment_new_timestamp` (el valor corregido, null si es anulación)
  - `adjustment_log_type`: IN | OUT
  - `adjustment_justification` (texto obligatorio, min 10 chars)
  - `adjustment_created_by` (FK a users)
  - `adjustment_created_at` (timestamp)
  - `adjustment_version`
- Migración: agregar a `vpg_payrolls`:
  - `payrolls_approved_by` (FK nullable a users)
  - `payrolls_approved_at` (timestamp nullable)
  - `payrolls_notes` (texto libre, notas del jefe)
- `npx prisma generate` exitoso

**Cobertura:** MARCAS-03, MARCAS-04, MARCAS-05, PLANILLA-04, PLANILLA-06

### Phase 33 — Backend: Motor de Marcas Efectivas + API de Ajustes
**Goal:** Crear el servicio de ajustes con CRUD completo y el motor que fusiona marcas originales con ajustes para producir las "marcas efectivas" que consume la planilla.

**Entregables:**
- `ClockLogAdjustmentService.ts`: métodos estáticos
  - `createAdjustment(data)` — crea ajuste (add/edit/void) + audit log atómico
  - `getAdjustmentsForEmployee(employeeId, startDate, endDate)` — ajustes en rango
  - `getAdjustmentHistory(clockLogId)` — historial de ajustes de una marca
- `EffectiveMarksService.ts`: el motor central
  - `getEffectiveMarks(employeeId, startDate, endDate)` — marcas originales con ajustes aplicados
  - `getEffectiveMarksByDay(employeeId, startDate, endDate)` — agrupadas por día con pares IN/OUT
  - `getEffectiveMarksForAllEmployees(startDate, endDate)` — batch para planilla (Map<employeeId, logs[]>)
- **REWIRE CRÍTICO: `NomineeService.preloadClockLogs`** — reemplazar la query directa a `vpg_clock_logs` por una llamada a `EffectiveMarksService.getEffectiveMarksForAllEmployees()`. Esto es lo que conecta las correcciones de marcas con el cálculo de planilla. Sin este cambio, corregir marcas no se refleja en el pago.
  - Solo marcas con status `valid` entran al cálculo
  - Ajustes `edit` reemplazan el timestamp original
  - Ajustes `void` excluyen la marca del cálculo
  - Ajustes `add` inyectan marcas nuevas
- Controller + Routes con auth y validación Zod
- Endpoint `GET /clock-logs/effective` — marcas efectivas agrupadas por empleado y día
- Endpoint `POST /clock-logs/adjustments` — crear ajuste
- Endpoint `GET /clock-logs/adjustments/:employeeId` — historial de ajustes

**Cobertura:** MARCAS-02, MARCAS-03, MARCAS-04, MARCAS-05, MARCAS-06, PLANILLA-02

### Phase 34 — Frontend: Rediseño Clock Logs (Vista Agrupada)
**Goal:** Reemplazar la tabla plana de marcas por una vista jerárquica: empleado → día → pares entrada/salida, con indicadores visuales de estado.

**Entregables:**
- Nuevo layout de `/pages/clock-logs`:
  - **Vista por empleado**: cards colapsables, un card por empleado
  - **Dentro del card**: tabla de días del período seleccionado
  - **Dentro del día**: pares IN/OUT con horas calculadas por par
  - **Indicadores**: 
    - Verde: día completo, marcas válidas
    - Amarillo: día con inconsistencias (marca huérfana, >10h)
    - Rojo: día sin marcas o con anomalía
    - Azul: día con correcciones aplicadas
    - Gris: día no laboral (domingo/feriado)
- **Resumen por empleado**: total horas período, días con problemas, días completos
- **Filtros**: rango de fechas, empleado específico, "solo días con problemas"
- **Empty state**: guía al jefe ("Importe marcas desde el archivo del reloj marcador para comenzar")
- Hook `useEffectiveMarks.ts` que consume el endpoint de marcas efectivas
- Servicio `effectiveMarksService.ts`

**Cobertura:** MARCAS-01, UX-01, UX-03

### Phase 35 — Frontend: Corrección de Marcas (Agregar/Editar/Anular)
**Goal:** Implementar los modales de corrección de marcas con justificación obligatoria y vista de auditoría.

**Entregables:**
- **Modal "Agregar marca faltante"**:
  - Selector de empleado (pre-llenado si se abre desde el card del empleado)
  - Selector de fecha (pre-llenado al día seleccionado)
  - Selector de hora (time picker)
  - Tipo: Entrada / Salida
  - Justificación (textarea, mínimo 10 caracteres)
  - Botón confirmar con preview: "Se agregará una marca de entrada para Juan Pérez el 15/04/2026 a las 08:00"
- **Modal "Editar marca"**:
  - Muestra valor original (no editable)
  - Campo para nuevo timestamp
  - Justificación obligatoria
  - Preview del cambio
- **Modal "Anular marca"**:
  - Muestra la marca a anular
  - Justificación obligatoria
  - Diálogo de confirmación: "¿Está seguro? Esta marca se marcará como anulada. El valor original se conserva."
- **Panel de historial de auditoría**:
  - Timeline de cambios por marca, expandible dentro del día
  - Muestra: quién, cuándo, qué cambió, justificación
- Todos los modales con `AnimatePresence` + `motion.div`
- Todos los formularios con `react-hook-form` + `zodResolver`

### Phase 36 — Backend: State Machine de Planilla + Aguinaldo
**Goal:** Implementar el ciclo de vida formal de planilla (BORRADOR → APROBADA → PAGADA) y refinar el cálculo de aguinaldo proporcional.

**Entregables:**
- `PayrollStateMachine.ts` — transiciones válidas con validación:
  - `BORRADOR` → `APROBADA` (requiere `approved_by`)
  - `APROBADA` → `PAGADA`
  - `APROBADA` → `BORRADOR` (re-apertura, genera audit log)
  - `BORRADOR` → `BORRADOR` (recálculo, guarda snapshot previo)
- Actualizar `PayrollService`:
  - `approvePayroll(payrollId, userId)` — valida estado, registra aprobación
  - `markAsPaid(payrollId)` — valida que esté aprobada
  - `reopenPayroll(payrollId, userId, reason)` — audit trail de re-apertura
  - `recalculatePayroll(payrollId)` — solo en BORRADOR, guarda versión anterior
- Actualizar `NomineeService.calculatePayrollForPeriod`:
  - Consumir `EffectiveMarksService` en lugar de marcas crudas
  - OT simplificado: horas sobre `employee_required_hours_biweekly` × 1.5
- Refinar aguinaldo:
  - `calculateAguinaldo(employeeId, periodStart, periodEnd)` — proporcional a meses trabajados
  - Fórmula: `(suma de salarios brutos en período) / 12`
  - Si menos de 12 meses: `suma / 12` (proporcional natural)
- Nuevos endpoints:
  - `POST /payroll/:id/approve`
  - `POST /payroll/:id/pay`
  - `POST /payroll/:id/reopen`
  - `POST /payroll/:id/recalculate`

**Cobertura:** PLANILLA-02, PLANILLA-04, PLANILLA-05, PLANILLA-06

### Phase 37 — Frontend: Wizard de Planilla Quincenal
**Goal:** Reemplazar la página actual de cálculo con un wizard guiado de 3 pasos que lleve al jefe desde la selección de período hasta la aprobación.

**Entregables:**
- **Paso 1 — Seleccionar período**:
  - Cards pre-calculados con los períodos quincenales disponibles (ej: "1-15 Abril 2026", "16-30 Abril 2026")
  - Indicador de estado: "Sin calcular", "Borrador", "Aprobada", "Pagada"
  - El jefe solo hace click en el período deseado
  - Opción de período personalizado colapsada por defecto
- **Paso 2 — Revisión de cálculo**:
  - Resumen general: total empleados, total horas, total bruto, total deducciones, total neto
  - Tabla por empleado expandible con desglose:
    - Horas regulares | Horas extra | Monto regular | Monto extra | Deducciones | Bonificaciones | Neto
  - Alertas por empleado (amarillo=advertencia, rojo=requiere atención):
    - "Juan Pérez: 3 días sin marcas en el período"
    - "María López: marca huérfana el 12/04 — ¿desea corregir antes de aprobar?"
  - Link directo a corrección de marcas desde la alerta
  - Botón "Recalcular" si se hicieron correcciones
- **Paso 3 — Confirmar y aprobar**:
  - Resumen ejecutivo (totales finales)
  - Notas opcionales del jefe
  - Diálogo de confirmación: "Al aprobar esta planilla, los montos quedarán registrados y no se podrán modificar sin re-abrirla. ¿Confirma la aprobación?"
  - Botón de aprobar con estado de carga
- **Lista de planillas**:
  - Actualizar vista existente con badges de estado (Borrador/Aprobada/Pagada)
  - Acciones contextuales según estado (aprobar, pagar, re-abrir, exportar)

### Phase 38 — Tests Unitarios + Verificación de Integración
**Goal:** Cubrir con tests la lógica crítica de negocio: motor de marcas efectivas, cálculo de planilla, aguinaldo, y state machine.

**Entregables:**
- Tests para `EffectiveMarksService`:
  - Marca original sin ajustes → se retorna tal cual
  - Marca con ajuste `edit` → retorna timestamp corregido
  - Marca con ajuste `void` → se excluye del resultado
  - Marca `add` sin original → se incluye como nueva
  - Múltiples ajustes sobre misma marca → último gana
- Tests para cálculo de planilla:
  - Período completo sin incidencias → horas y pago correctos
  - Empleado con horas extra → cálculo al 1.5×
  - Empleado sin marcas un día → inconsistencia reportada
  - Deducciones fijas y porcentuales aplicadas correctamente
- Tests para aguinaldo:
  - Empleado 12 meses completos → promedio de 12 salarios
  - Empleado 6 meses → proporcional
  - Empleado sin planillas en período → monto 0
- Tests para state machine:
  - Transiciones válidas permitidas
  - Transiciones inválidas rechazadas (ej: PAGADA → BORRADOR)
  - Re-apertura genera audit log
- Verificación de integración end-to-end:
  - Flujo completo: importar marcas → corregir → calcular → aprobar

**Cobertura:** PLANILLA-02, PLANILLA-05, MARCAS-06
