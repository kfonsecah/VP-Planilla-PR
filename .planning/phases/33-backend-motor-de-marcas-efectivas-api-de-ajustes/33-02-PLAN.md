  <files>src/backend/src/service/ClockLogEffectiveService.ts</files>
  <behavior>
    - `getEffectiveLogs(employeeId, startDate, endDate)`:
      1. Fetch all `vpg_clock_logs` for employee/range (includes manual ADD logs naturally).
      2. Fetch latest ACTIVE adjustments (EDIT/VOID only) for these log IDs:
         - Use `findMany` with `adjustment_clock_log_id: { in: logIds }` and `adjustment_status: ACTIVE`.
         - orderBy `adjustment_created_at DESC` — engine picks the most recent per log.
         - Do NOT use Prisma.distinct — it picks arbitrary row per group in PostgreSQL. Instead: fetch all ACTIVE adjustments then reduce to latest per log_id in JS.
      3. For each log:
         - If latest adjustment type is VOID: exclude from result.
         - If latest adjustment type is EDIT: use `adjustment_new_timestamp` as effectiveTimestamp, `clock_logs_timestamp` as originalTimestamp.
         - If no active adjustment: effectiveTimestamp = originalTimestamp = `clock_logs_timestamp`.
      4. ADD logs are already in `vpg_clock_logs` (source=manual) — no special handling needed.
  </behavior>
  <action>
    Implement `ClockLogEffectiveService` with static methods:
    - `getEffectiveLogs(employeeId: number, startDate: Date, endDate: Date): Promise<EffectiveMark[]>`
    - `EffectiveMark` interface: `{ id, employeeId, originalTimestamp, effectiveTimestamp, logType: ClockLogType, adjustmentType: 'EDIT' | 'VOID' | 'NONE', source }`
    - `getEffectiveMarksForAllEmployees(startDate: Date, endDate: Date): Promise<Map<number, EffectiveMark[]>>`
      — batch version for payroll: fetches all logs in range, all active adjustments in one query, groups result by employeeId.
      — this is what NomineeService.preloadClockLogs will call in Phase 33 Task 2 of Plan 03.
    - Import enums from `@prisma/client`: `ClockLogAdjustmentType`, `ClockLogAdjustmentStatus`, `ClockLogType`.
  </action>
